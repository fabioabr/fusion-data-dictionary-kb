03 — Descrever Tabelas

.md Skeleton → .md Enriquecido (LLM)
LLM Opus
~$0.03/tabela
~2h para 1.205 tabelas
Objetivo
Esta é a etapa mais rica em conteúdo do pipeline. Um LLM (Claude Opus) lê cada .md skeleton gerado na etapa 01 e o enriquece com: parágrafo de visão geral do negócio, descrição para cada coluna, categoria semântica (PK FK Amount Date Status Audit Config), seção de relacionamentos com wikilinks para tabelas pai/filha e colunas FK, exemplos de consultas SQL típicas, e nível de confiança por coluna.
DOR — Definition of Ready
Pré-requisitos para executar esta etapa
  • Arquivos .md skeleton existem a partir da etapa 01 em src/kb/.../docs/{MOD}/tables/
  • API key Anthropic configurada (ou sessão ativa no Claude Code)
  • Contexto do módulo compreendido (o que cada tabela faz no Oracle Fusion)
DOD — Definition of Done
Critérios de aceite para esta etapa
  • Todo .md possui parágrafo de Visão Geral (2-4 sentenças descrevendo o propósito de negócio)
  • Todas as colunas possuem descrição preenchida em pt-BR
  • Categorias semânticas atribuídas a cada coluna (PK, FK, Amount, Date, Status, Audit, Config)
  • Relacionamentos com colunas FK identificados e wikilinks apontando para tabelas corretas
  • Pelo menos 2 exemplos de consultas SQL por tabela
  • Níveis de confiança marcados por coluna
  • Nenhum placeholder "a ser preenchido" restante
Entrada
Arquivos de Entrada
Arquivo Caminho Descrição
Skeletons src/kb/oracle-fusion-data-dictionary/docs/{MOD}/tables/*.md Arquivos .md skeleton gerados na etapa 01 — contêm front matter, lista de colunas sem descrição e seções vazias
Saída
Arquivos Enriquecidos

Os mesmos arquivos .md são enriquecidos in-place. Exemplo de antes/depois de uma linha de coluna:

ANTES (skeleton da etapa 01)
| 1 | LEDGER_ID | NUMBER(15) | NOT NULL | — | — | — | — |
DEPOIS (enriquecido pela LLM)
| 1 | LEDGER_ID | NUMBER(15) | NOT NULL | PK.FK | Identificador do ledger (livro contábil) | [[gl_ledgers]] | 95% |

Exemplo de seções adicionadas ao .md:

## Visão Geral Tabela central do módulo General Ledger que armazena os saldos contábeis por período, ledger e combinação de contas. Utilizada em relatórios de balancete, DRE e balanço patrimonial. ## Relacionamentos ### Tabelas-pai (FK de entrada) - [[gl_ledgers]] — via coluna `LEDGER_ID` - [[gl_code_combinations]] — via coluna `CODE_COMBINATION_ID` ### Tabelas-filha (FK de saída) - [[gl_je_lines]] — referencia `BALANCE_ID` ## Uso Típico ### Consultas SQL Exemplo ```sql -- Saldo por período e ledger SELECT PERIOD_NAME, SUM(BEGIN_BALANCE_DR - BEGIN_BALANCE_CR) AS SALDO FROM GL_BALANCES WHERE LEDGER_ID = :ledger_id GROUP BY PERIOD_NAME; ```
Como Funciona
Passo a passo do processo de enriquecimento
  1. Lê o arquivo .md skeleton (front matter + tabela de colunas + seções vazias)
  2. Envia ao LLM com prompt de contexto: módulo, nome da tabela, lista de colunas e domínio Oracle Fusion
  3. LLM retorna: visão geral, descrições de colunas, categorias semânticas, relacionamentos e exemplos SQL
  4. Faz o parse da resposta do LLM e atualiza as seções do .md
  5. Valida: nenhuma descrição vazia, categorias são válidas, wikilinks usam nomes corretos de tabelas
  6. Grava o .md enriquecido no mesmo caminho de entrada
Prompt Engineering
Estrutura do prompt enviado ao LLM

O prompt é construído dinamicamente para cada tabela e contém os seguintes blocos:

  • Nome da tabela — ex: GL_BALANCES
  • Contexto do módulo — ex: "General Ledger do Oracle Fusion Cloud — módulo de contabilidade"
  • Lista de colunas — nomes e tipos de dados extraídos do Excel
  • Instruções de domínio — conhecimento específico do Oracle Fusion (naming conventions, padrões de FK)
  • Inferência de FK — colunas *_ID são inferidas como FK para a tabela-pai correspondente (ex: LEDGER_ID → GL_LEDGERS)
  • Formato de saída — template Markdown esperado com seções, tabela de colunas e wikilinks
Relacionamentos são inferidos por convenção de nomenclatura: colunas terminadas em _ID sugerem FK para a tabela-pai. Ex: INVOICE_ID → [[ap_invoices_all]].
Validação de Qualidade
Verificações pós-geração

Após a geração pelo LLM, uma etapa de revisão verifica:

  • Tabelas alucinadas — nomes de tabelas nos relacionamentos devem existir em table-mappings.txt
  • Colunas FK válidas — colunas referenciadas como FK devem existir na lista de colunas da tabela
  • Idioma correto — descrições devem estar em pt-BR
  • Confiança realista — níveis de confiança devem refletir a certeza real da inferência
  • Sem placeholders — nenhum campo com "a ser preenchido" ou "—" pode permanecer
Uso
Modos de Execução

A etapa pode ser executada de duas formas:

# Modo interativo — via sessão Claude Code # Abrir sessão e solicitar enriquecimento tabela por tabela claude "Enriqueça o arquivo src/kb/.../docs/GL/tables/gl_balances.md" # Modo batch — via script com API Anthropic cd Pipeline/03-generate-table-descriptions/ python enrich_tables.py --module GL # Batch completo (todos os módulos) python enrich_tables.py --all # Re-enriquecer tabelas específicas python enrich_tables.py --module AP --force --tables ap_invoices_all,ap_checks_all
Notas Importantes
Esta é a etapa mais cara do pipeline em termos de custo de API (~$0.03 por tabela). Planeje o orçamento antes de rodar em batch para todos os módulos.
Relacionamentos sem colunas FK reais são inválidos e devem ser removidos. Em nossa última limpeza, 455 relacionamentos inválidos foram removidos — o LLM havia inventado "relacionamentos lógicos" sem FK real.
O LLM às vezes inventa relacionamentos lógicos que não possuem colunas FK no schema. Esses devem ser revisados e removidos na etapa de validação.
Descrições são geradas em pt-BR. A tradução para EN/ES ocorre em etapa separada do pipeline.
