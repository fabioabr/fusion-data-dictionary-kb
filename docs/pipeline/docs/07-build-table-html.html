07 — Build HTML Tabelas

.md + .i18n.json + template → HTML standalone
Script Python
Sem LLM
~15s para 1.045 tabelas
Objetivo
Esta etapa gera páginas HTML self-contained (standalone) a partir dos arquivos .md enriquecidos e seus respectivos .i18n.json. Cada tabela recebe uma página HTML individual com 4 abas: Overview (visão geral, descrição, stats), Columns (tabela de colunas com busca), Relationships (tabelas pai/filha com cross-links) e SQL (consultas exemplo e filtros comuns). O HTML é gerado usando um template (template_system_doc.html) com CSS e JS inline, sem dependências externas.
DOR — Definition of Ready
Pré-requisitos para executar esta etapa
  • Arquivos .md enriquecidos (com descrições) existem — gerados na etapa 03
  • Arquivos .i18n.json existem para cada tabela — gerados na etapa 06 (tradução)
  • Template template_system_doc.html existe no diretório de templates
  • Arquivo table-mappings.txt disponível para resolução de cross-links entre tabelas
DOD — Definition of Done
Critérios de aceite para esta etapa
  • Um arquivo .html por tabela criado em docs/fusion-kb/data-dictionary/{MOD}/tables/
  • HTML é standalone self-contained — CSS e JS inline, sem dependências externas (exceto Google Fonts e Remix Icon CDN)
  • Tema dark/light com toggle funcional, header e footer com gradiente fixo escuro
  • Conteúdo em 3 idiomas (PT-BR, EN, ES) via data-i18n + dicionário JS
  • Cross-links entre tabelas resolvidos como caminhos relativos (incluindo cross-module)
Entrada
Arquivos de Entrada
Arquivo Caminho Descrição
.md src/kb/oracle-fusion-data-dictionary/docs/{MOD}/tables/{table}.md Markdown enriquecido com front matter, colunas, descrições, relacionamentos e SQL (etapa 03)
.i18n.json src/kb/oracle-fusion-data-dictionary/docs/{MOD}/tables/{table}.i18n.json Dicionário i18n com traduções PT-BR, EN e ES para todos os textos (etapa 06)
Template Pipeline/templates/template_system_doc.html Template HTML com placeholders para injeção de conteúdo e dicionário i18n
Mapeamento src/raw-files/table-mappings.txt Mapeamento tabela → módulo para resolução de cross-links entre módulos
Saída
Arquivos Gerados

Padrão de saída por tabela:

docs/fusion-kb/data-dictionary/{MOD}/tables/{table_name}.html

Cada HTML contém 4 abas:

Aba Conteúdo
Overview Visão geral da tabela: descrição, stats grid (colunas, PKs, FKs, nullable), módulo e sistema
Columns Tabela completa de colunas com busca/filtro: #, nome, tipo, nullable, categoria, descrição, FK, confiança
Relationships Tabelas-pai (FK de entrada) e tabelas-filha (FK de saída) com cross-links clicáveis
SQL Consultas SQL exemplo e filtros comuns — snippets prontos para uso
Template & Design System
Padrões visuais do HTML gerado
  • Tipografia: Poppins (Google Fonts) — nenhuma outra fonte utilizada
  • Ícones: Remix Icon (CDN) — nenhuma outra biblioteca de ícones
  • CSS e JS inline — arquivo self-contained, sem frameworks externos
  • Header e footer com gradiente escuro fixo (#1a2332 → #15202e → #1a2838) em ambos os temas
  • Internacionalização via atributo data-i18n + dicionário JS embutido no HTML
  • Toggle dark/light theme com persistência em localStorage
  • Seletor de idioma com bandeiras SVG (PT-BR, EN, ES) no header
Cross-links
Resolução de wikilinks para caminhos relativos

Os arquivos .md usam wikilinks ([[table_name]]) para referenciar outras tabelas. O script resolve esses links para caminhos relativos HTML usando o arquivo table-mappings.txt:

# Mesmo módulo (GL → GL) [[gl_ledgers]] → gl_ledgers.html # Cross-module (GL → AP) [[ap_invoices_all]] → ../../AP/tables/ap_invoices_all.html # Tabela não encontrada — link desabilitado [[tabela_inexistente]] → <span class="broken-link">tabela_inexistente</span>
Uso
Comandos de Execução

Execute o script a partir do diretório raiz do Pipeline:

# Gerar HTMLs de todas as tabelas (batch) python build_table_html.py --batch # Gerar HTMLs apenas de um módulo python build_table_html.py --module GL # Forçar re-geração (sobrescreve HTMLs existentes) python build_table_html.py --batch --force
Notas Importantes
Esta etapa NÃO usa LLM — é um script Python determinístico que combina dados do .md e .i18n.json com o template HTML.
Cross-links quebrados (tabelas não encontradas no table-mappings.txt) são renderizados como texto sem link, com classe broken-link.
O template usa design tokens CSS (var(--*)) para garantir consistência visual — qualquer alteração no playground propaga automaticamente.
Tabelas sem arquivo .i18n.json serão geradas apenas em PT-BR. O seletor de idioma aparece, mas EN e ES exibirão o conteúdo em português.
