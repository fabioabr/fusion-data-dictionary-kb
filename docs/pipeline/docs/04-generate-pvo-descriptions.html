04 — Descrever PVOs

JSON básico → JSON com descrição de negócio (LLM)
LLM Opus
~200 PVOs/min com agentes paralelos
Objetivo
Esta etapa utiliza o LLM Claude Opus para gerar descrições de negócio em pt-BR (1-2 frases) para cada PVO (Parameterized View Object) do Oracle Fusion. O modelo analisa o pvo_name, o caminho AM (Application Module), as tabelas base e os atributos-chave de cada PVO para produzir uma descrição concisa e significativa. O campo meta.description de cada JSON é atualizado com texto descritivo de até 300 caracteres.
DOR — Definition of Ready
Pré-requisitos para executar esta etapa
  • Arquivos JSON dos PVOs gerados na etapa 02 existem em src/kb/.../docs/{MOD}/pvos/
  • Cada JSON contém campos pvo_name, am_path, tables e key_attributes preenchidos
  • Acesso à API Anthropic (Claude Opus) configurado via Claude Code
DOD — Definition of Done
Critérios de aceite para esta etapa
  • Campo meta.description preenchido com texto em pt-BR para todos os PVOs
  • Descrições com 1-2 frases significativas (máximo 300 caracteres)
  • Nenhum PVO com descrição vazia, genérica ou placeholder
  • Termos técnicos Oracle mantidos em inglês (PVO, View Object, Application Module)
Entrada
Arquivos de Entrada
Arquivo Caminho Descrição
JSON src/kb/.../docs/{MOD}/pvos/{pvo_name}.json Arquivos JSON individuais por PVO, gerados na etapa 02, com estrutura, tabelas e atributos
Saída
Arquivos Atualizados

O mesmo arquivo .json é atualizado in-place com o campo meta.description preenchido:

Antes (saída da etapa 02):

{ "pvo_name": "BenefitPersonBenefitBalancePVO", "am_path": "oracle.apps.hcm.benefits.core.eligibility.publicModel.applicationModule", "tables": ["BEN_PER_BNFT_BAL"], "key_attributes": ["PERSON_ID", "BENEFIT_BALANCE_ID", "PLAN_TYPE_ID"], "meta": { "description": "" } }

Depois (saída da etapa 04):

{ "pvo_name": "BenefitPersonBenefitBalancePVO", "am_path": "oracle.apps.hcm.benefits.core.eligibility.publicModel.applicationModule", "tables": ["BEN_PER_BNFT_BAL"], "key_attributes": ["PERSON_ID", "BENEFIT_BALANCE_ID", "PLAN_TYPE_ID"], "meta": { "description": "Saldos de benefícios por colaborador, rastreando os valores acumulados e utilizados por tipo de plano de benefício no módulo HCM Benefits." } }
Como Funciona
Passo a passo do processo
  1. Consolida todos os PVOs de um módulo em uma lista única com pvo_name, am_path, tables e key_attributes
  2. Divide a lista em chunks de ~200 PVOs para processamento paralelo
  3. Envia cada chunk para um agente Opus com prompt estruturado: "Gere uma descrição de negócio em pt-BR (1-2 frases, máx. 300 chars) para cada PVO"
  4. Cada agente retorna um JSON com pares {pvo_name: description}
  5. Os resultados dos agentes são consolidados e validados (sem descrições vazias ou duplicadas)
  6. As descrições são escritas de volta nos arquivos JSON individuais, atualizando o campo meta.description
Estratégia de Paralelização
Agentes Opus em Paralelo

Para maximizar a velocidade, o processo utiliza 4-8 agentes Opus simultaneamente, cada um processando um chunk de 100-200 PVOs. Essa estratégia reduz o tempo total de processamento de ~40 minutos (sequencial) para ~3-5 minutos.

Exemplo para HCM (841 PVOs): divididos em 5 chunks de ~168 PVOs, processados por 5 agentes em paralelo. Cada agente leva ~3 minutos para gerar todas as descrições do seu chunk, totalizando ~3-5 minutos de tempo de parede.

A paralelização é feita via Claude Code Agent tool — cada agente é uma chamada independente ao LLM, permitindo processamento verdadeiramente paralelo sem contenção de recursos.
Uso
Comandos de Execução

Execute via Claude Code com a ferramenta Agent:

# Via Claude Code — descrever PVOs de um módulo # (dentro do Claude Code, usar Agent tool com prompt estruturado) # Prompt para o agente: # "Leia todos os PVOs em src/kb/.../docs/HCM/pvos/, # gere descrições de negócio em pt-BR para cada um, # e atualize o campo meta.description nos JSONs." # Módulos disponíveis: GL, AP, AR, PO, HCM
